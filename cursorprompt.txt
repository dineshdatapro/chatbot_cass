# Multi-Tenant Agentic RAG SaaS - Behaviour & Decision Flow Improvements

This project is a **multi-tenant Agentic RAG SaaS platform** where every tenant uploads their own documents and gets an AI chatbot that can be embedded into their website.

The overall architecture is already working correctly (FastAPI, LangGraph, Qdrant, Parent-Child Retrieval, Multi-Tenant Isolation, Authentication, Widget, etc.).

**Do NOT redesign or rewrite the architecture.**

Instead, review the entire reasoning flow and improve the chatbot's behaviour so it feels like a professional SaaS product similar to Chatbase, Intercom AI, Zendesk AI, or Microsoft Copilot Studio.

---

# Primary Goals

Improve:

* Planner behaviour
* Query understanding
* Query rewriting
* Retrieval decision logic
* Intent classification
* Final answer generation
* Fallback behaviour
* Conversation experience

Do NOT change:

* FastAPI architecture
* LangGraph workflow
* Qdrant integration
* Parent-child retrieval
* Multi-tenant architecture
* APIs
* Authentication

# Current Problems

## Problem 1

Example:

User:

Current:

Please specify the context.

Expected:

Automatically infer that the user is asking about the current tenant's organization and retrieve the relevant information.

Never ask unnecessary clarification questions.

## Problem 3

Example:

Current:

The term is unclear.

Expected:

Attempt lightweight typo correction.

Normalize the query.

Perform retrieval.

Only reject the query if retrieval genuinely cannot identify the intent.

## Problem 4

Example:

Current:

"I couldn't find any information..."

This is incorrect.

Capability questions should NEVER enter the RAG pipeline.

---

## Problem 5

During reasoning I can see that relevant information is retrieved, but the final response still becomes:

"I couldn't find any information..."

Investigate the complete LangGraph workflow.

Check whether:

* planner node
* validator node
* confidence check
* answer generation node

are discarding valid retrieved context before generating the final response.

If retrieved context is sufficient, never replace it with a generic fallback.

# Multi-Tenant Behaviour

Every tenant uploads different documents.

Never hardcode company names.

Never hardcode products.

Never hardcode courses.

Never hardcode organization details.

Instead, dynamically infer everything from the tenant's uploaded knowledge.

Examples:

Tenant A

ABC Hospital

↓

Answer like ABC Hospital assistant

Tenant B

XYZ College

↓

Answer like XYZ College assistant

Tenant C

DataPro

↓

Answer like DataPro assistant

The chatbot must automatically adapt.

---

# Tenant Isolation

Verify that every retrieval uses the tenant context.

Ensure:

* tenant_id filtering
* tenant-specific vector collection or namespace
* tenant-specific conversations
* tenant-specific document lookup
* tenant-specific caching

Never leak information across tenants.

# Clarification Policy

Ask clarification ONLY when:

The question itself is unclear.

Examples:

"It"

"That"

"This"

"Tell me more"

with no conversation context.

Do NOT ask clarification because retrieval returned no results.

---

# Conversation Memory

Support follow-up questions.

Example:

What courses are available?

↓

What is the duration?

↓

What are the fees?

↓

Do they provide placements?

The chatbot should use previous conversation context before requesting clarification.

---

# Fallback Behaviour

If relevant information exists:

Answer naturally.

If information does not exist:

Do NOT hallucinate.

Do NOT expose internal implementation.

Never mention:

* uploaded documents
* knowledge base
* RAG
* embeddings
* vector search
* retrieval
* chunks
* Qdrant

Instead respond naturally.

Examples:

"I'm sorry, but I don't have enough information to answer that accurately."

"I can't confidently answer that at the moment."

"I'm not able to verify that information."

"I don't have reliable information about that."

Choose the most natural response based on the conversation.

---

The assistant should be:
